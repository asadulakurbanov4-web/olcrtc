## 0. Как читать документ

Контекст целевого проекта (из `OlcPanel_00..08` + реального кода):

- **Одна общая комната + один общий ключ** шифрования (отказ от «комната на клиента» — было нестабильно).
- **Блокировка через `deviceId` (`x-hwid`)**: панель пишет `authz.json` через `authz_writer.write_authz_file(db)`, сервер `internal/authz/authz.go` (`Gate.Allowed(deviceID)`) принимает решение. Сейчас `authz.mode: "off"`.
- **Реальное поведение `Gate`** (важно для всего раздела 4): схема файла `{version, updated_at, allow[], deny[]}`, режимы `allowlist`/`denylist`/`off`; перезагрузка по **mtime-кэшу** (ленивая, на каждом `Allowed()` под мьютексом + sweep живых сессий по `EnforceInterval`). **Поведение при сбое чтения — РЕШЕНО (5.2): `last-known-good` (`authz.fail_mode: lkg`)** — при отсутствии/битом/непарсящемся файле `Gate` держит предыдущий валидный allowlist, а не открывается всем. Прежний `fail-open` (`Allowed()→true` при ошибке) остаётся только как явный диагностический режим `authz.fail_mode: open` для отладки/free-инсталляций.
- **Стек панели:** Python 3.11 · FastAPI · SQLite · Vanilla JS · Nginx.
- **Модель данных уже заложена:** `users/clients`, `subscriptions`, `user_devices(device_id, access_token, server_id, last_used_at)`, `servers`, `plans`, `payments`, `traffic_daily/snapshots`, `server_settings(warn_days_before)`, `audit_log/activity_logs`.
- **Вектор:** MVP на одном сервере → горизонтальное масштабирование (50–100 клиентов/VPS, одна центральная панель, `MultiServerAuthzCoordinator`).
- **Трафиковый аккаунтинг де-приоритизирован** (WebRTC-туннель не отдаёт Xray-статистику).

**Шкала оценок (1–10):**

| Критерий | 1 | 10 |
|---|---|---|
| **Полезность** | бесполезно | критично ценно |
| **Необходимость** | можно никогда | без этого не продать/не масштабировать |
| **Сложность внедрения** | пара часов | большой архитектурный кусок |
| **Применимость к архитектуре** (одна комната + deviceId) | противоречит модели | ложится идеально |

«Хорошая» фича = высокая Полезность/Применимость при **низкой** Сложности.

## 1. Общий вывод: какая панель полезнее и почему

**Marzban — основной донор (≈75% полезных идей).** Тот же стек (Python/FastAPI/SQLAlchemy + REST/Swagger + JWT) и, главное, **та же ментальная модель будущего OlcPanel** — «одна центральная панель + флот периферийных серверов» (Marzban-node). Из Marzban берём модель multi-server, health-check нод, уведомления (Telegram + Webhook), напоминания об истечении, шаблоны/страницу подписки, авто-отключение по сроку, RBAC администраторов.

**3x-ui — донор «операционных мелочей» (≈25%).** Go + inbound-центричная модель архитектурно дальше; **управление inbounds/протоколами не применимо вообще**. Но у 3x-ui хорошо сделаны вещи, не зависящие от inbound-модели: **безопасность входа в панель** (2FA, секретный путь, fail2ban), **авто-бэкап в Telegram**, **мониторинг состояния системы**, **cron-очистка просроченных**, **просмотр онлайн-клиентов**.

**Принцип фильтрации:** OlcRTC не имеет inbounds, протоколов Xray, per-client сетевого аккаунтинга и IP-маршрутизации. Всё, что завязано на Xray-инбаунды/протоколы/трафик-лимиты/IP-роутинг, отсекаем сразу. Идентичность клиента — `deviceId`, а не пара (inbound, ключ).

**Сквозной приоритет (исправление ревью):** горизонтальное масштабирование — это **не фича Этапа 2–3, а архитектурный фундамент**. Реализация мульти-push может включаться позже, но **контракт координации `authz.json` (ServerRegistry, `server_id`, `MultiServerAuthzCoordinator`, версионирование payload) проектируется и закладывается в схему с Этапа 1**, иначе придётся ломать модель данных и writer на ходу. Раздел 4 — основной в этом документе.

## 2. Функции с высоким приоритетом (Внедрять)

### 2.1 Авто-отключение по истечению подписки (Cron → authz.json)
- **Откуда:** оба (Marzban — review-jobs по `expire`; 3x-ui — Cron на удаление просроченных).
- **Описание:** периодическая джоба сканирует `subscriptions.expires_at`; для просроченных переводит `user_devices.is_active=false` и вызывает `authz_writer.write_authz_file(db)` → сервер перестаёт пускать `deviceId`. **Soft-block, без удаления.**
- **Оценки:** Полезность 10 · Необходимость 10 · Сложность 3 · Применимость 10.
- **Рекомендация:** **Внедрять.** **Этап 1.**
- **Обоснование:** это и есть монетизация продукта («продал на срок → по истечении отрубилось само»). Ложится на существующий Phase 1. Делать `soft block` через `authz.json`, **не** копировать авто-удаление. ⚠️ Запись должна быть атомарной (см. 5.1): при `fail_mode: lkg` (5.2) битый файл больше не «пустит всех», но неатомарная запись всё равно может на миг «потерять» allow или зафиксировать неполный срез — атомарность остаётся обязательной.

### 2.2 Журнал действий администратора (аудит)

- **Описание:** запись `create_client / block_client / generate_link / write_authz / authz_push / extend_subscription` в `audit_log`/`activity_logs` (таблицы уже есть).
- **Оценки:** Полезность 7 · Необходимость 6 · Сложность 2 · Применимость 10.

- **Обоснование:** дёшево; критично для отладки multi-server (без аудита push'ей отладка несогласованности невозможна — см. 5.3). Это не «приятно», а предпосылка раздела 4.

### 2.3 REST API + OpenAPI/Swagger
- **Откуда:** оба.
- **Описание:** у проекта уже FastAPI → включить авто-Swagger, описать контракты CRUD клиентов/подписок/устройств/authz/серверов.

### 2.4 Telegram-уведомления администратору
- **Откуда:** Marzban (богатая модель событий) + 3x-ui (нагрузка/бэкап).
- **Описание:** бот шлёт владельцу события: создан клиент, подписка продлена, клиент заблокирован по сроку, подписка истекает через N дней, **ошибка записи/доставки authz (push failed на server_id)**, сервер недоступен.

- **Описание:** та же Cron-джоба, что в 2.1, но за `server_settings.warn_days_before` дней до `expires_at` шлёт напоминание.

## 4. Горизонтальное масштабирование: архитектура координации authz

> Это центральный раздел документа. Ниже — не «идеи», а контракт, который должен быть заложен в модель данных и сервисы **сейчас**, чтобы переход single-server → multi-server не потребовал переписывания. Раздел согласован с `olcrtc-panel-architecture.md` (варианты A/B/C, `MultiServerAuthzCoordinator`) и уточнён реальным поведением `Gate`.

### 4.1 Что мы масштабируем и почему это про authz, а не про «много панелей»

Целевая модель — **одна центральная панель (источник истины в SQLite) + N дешёвых olcrtc-серверов** (50–100 клиентов каждый, свой `server.yaml`, свой `authz.json`, свой `Gate`). Клиент закрепляется за конкретным сервером (`server_id`) в момент выдачи подписки. Панель не маршрутизирует трафик — она **доставляет каждому серверу его срез allowlist** и следит, что он применился.

Следовательно, вся сложность масштабирования сводится к одному вопросу: **как надёжно и согласованно доставить актуальный `authz.json` (срез по `server_id`) на каждый сервер и убедиться, что `Gate` его прочитал.** Это и проектируем.

### 4.2 Базовый поток данных (single-server, как сейчас)

```
[событие: create/extend/block]
        │
        ▼
 user_devices / subscriptions  (SQLite, источник истины)
        │
        ▼
 authz_writer.write_authz_file(db)   ──атомарно──▶  /opt/olcrtc-panel/data/authz.json
                                                          │  (mtime изменился)
                                                          ▼
                                              Gate.refresh() по mtime-кэшу
                                                          ▼
                                              Gate.Allowed(deviceID)  → пускать/нет
```

Ключевое из реального кода: `Gate` перечитывает файл **только если изменился mtime**; при ошибке чтения/парсинга — **fail-open**. Эти два факта определяют все требования к доставке (4.5).

### 4.3 Варианты синхронизации (для флота 3–10 серверов)

Согласовано с `olcrtc-panel-architecture.md`. По сути это **адаптация модели Marzban-node** (центр → ноды), но через файл/HTTP вместо gRPC.

| Вариант | Механизм | Сложность | Окно несогласованности | Когда |
|---|---|---|---|---|
| **A. rsync + trigger** | Панель генерирует payload, `rsync`'ит на сервер, дёргает мелкий trigger (HTTP/systemd-path), который кладёт файл **атомарно** (`tmp`+`os.rename`) туда, где читает `Gate`. | Низкая | секунды–десятки секунд | **Этап 1–начало 2**, пока < 5–6 серверов |
| **B. Signed HTTP push** | Каждый сервер экспонирует защищённый (`mTLS`/подписанный токен) endpoint `/internal/authz/update`. Панель шлёт JSON+подпись; сервер **проверяет подпись, отвергает старую `version`, пишет атомарно, отвечает 200 только после успешной записи**. | Средняя | секунды | **5+ серверов** |
| **C. Online-проверка / короткоживущие токены** | `Gate` на handshake спрашивает панель (или валидирует подписанный device-токен). Файл перестаёт быть источником истины. | Высокая | ≈0 | долгосрочно, когда файловый подход станет болезненным |

**Почему именно так (а не «единый сетевой storage»):** общий storage (NFS/etcd/Redis) на каждый дешёвый VPS добавляет инфраструктурную зависимость и новую точку отказа, противоречит цели «дешёвые независимые VPS». Файл + локальный `Gate` **сознательно** выбран ради отказоустойчивости: упала панель — серверы продолжают пускать по последнему `authz.json`. Вариант C жертвует именно этим (панель становится SPOF, 5.4), поэтому он последний.

### 4.4 Компоненты центральной панели

```
            ┌───────────────────────────────────────────────┐
            │              Central OlcPanel (FastAPI)         │
            │                                                 │
   событие  │   IssuancePolicy ──выбирает server_id──┐        │
  ────────▶ │                                        ▼        │
            │   ServerRegistry(servers)        MultiServerAuthzCoordinator
            │      ▲ health-poll                      │ on_change(user, server_id)
            │      │                                  ▼
            │   ServerHealthService          _compute_allow_for_server(server_id)  ← SQLite
            │      │                                  │  payload{version,updated_at,allow,deny}
            │      │                                  ▼
            │      │                          ServerAuthzPusher (вариант A→B)
            │      │                                  │ push(server_id, payload)
            │      │                                  ▼  + log в activity_logs, alert при fail
            └──────┼──────────────────────────────────┼─────┘
                   │ GET /health                       │ rsync/HTTPS
                   ▼                                   ▼
        ┌──────────────────┐                ┌──────────────────┐
        │  olcrtc-server #1│ ... (N)        │  olcrtc-server #k │
        │  Gate + authz.json│                │  Gate + authz.json│
        └──────────────────┘                └──────────────────┘
```

- **`ServerRegistry`** — CRUD `servers`, статус (`active/maintenance/offline`), `priority`, `api_base`.
- **`IssuancePolicy`** — **единственный**, кто решает, на какой `server_id` посадить клиента (по `priority`/загрузке/региону). Координатор этого НЕ делает.
- **`MultiServerAuthzCoordinator`** — по событию (изменение подписки/устройства/ручной блок) **пересчитывает allow/deny для конкретного `server_id` из БД** (не хранит глобальный allowlist), вызывает pusher, логирует результат, при устойчивой ошибке — алерт + ручной re-push из админки. Режим **best-effort**: не дошло до одного — остальные работают.
- **`ServerAuthzPusher`** — абстракция доставки; реализация A (rsync+trigger) → B (signed HTTP). Скрывает транспорт от координатора.
- **`ServerHealthService`** — периодический poll `/health` каждого сервера: служба жива, какая `version`/`updated_at` у применённого `authz.json`, сколько `allow` загрузил `Gate`. Это и есть детектор расхождений (5.3).

**Скелет на Этапе 1:** все эти компоненты существуют, но `ServerAuthzPusher` имеет реализацию `LocalFilePusher` (просто вызывает текущий `authz_writer` для единственного дефолтного сервера). `Coordinator.on_change` вызывается, но проходит no-op-путь. Так контракт зафиксирован, а поведение не меняется. Включение реального push — переключением реализации pusher + feature-flag `multi_server_authz`.

### 4.5 Требования к payload и записи (выводятся из поведения `Gate`)

Эти требования — прямое следствие fail-open + mtime-кэша; они **обязательны** уже на Этапе 1, иначе мульти-сервер позже даст тихие баги.

1. **Атомарная запись.** Только `write tmp → fsync → os.rename`. Частичная запись валидного JSON может «потерять» allow (клиенты отвалятся), частичная запись битого JSON → **fail-open** (пустим всех, включая неоплативших). Вариант B усиливает это: «200 только после успешной записи».
2. **Монотонная версия.** В payload уже есть `version`/`updated_at`. На варианте B сервер **обязан отвергать payload со `version` ≤ применённой** — иначе ретраи/гонки доставки могут «откатить» allowlist на старый. (Текущий `Gate` берёт «последний по mtime» — для rsync-варианта A это значит: писать строго возрастающим mtime; для B — добавить version-check в endpoint.)
3. **Гарантия сдвига mtime.** `Gate` сравнивает `mtime.Equal`; два обновления в пределах гранулярности mtime ФС (на части ФС — 1 с) → второе может быть проигнорировано до следующего изменения. Writer/trigger должен явно «двигать» mtime (`os.utime`) или дедуплицировать частые записи.
4. **Подтверждение применения.** Доставка не считается успешной, пока health-poll (4.4) не увидел нужную `version` на сервере. Без этого «push прошёл» ≠ «правило применилось».
5. **Идемпотентность.** Координатор всегда **пересчитывает** срез из БД, никогда не «дельтит» поверх неизвестного состояния — повторная доставка безопасна.

### 4.6 Эволюция модели данных под `server_id` (детально, по ревью)

Где живёт `server_id` — **в обоих**, и это осознанно, не дублирование:

- **`subscriptions.server_id`** — «на каком сервере *продан* доступ». Заполняется `IssuancePolicy` в момент выдачи/продления. Это коммерческая привязка (клиент купил доступ к региону/серверу).
- **`user_devices.server_id`** — «на какой сервер *доставлять* allowlist для этого устройства». Обычно совпадает с подпиской, но отделено, потому что: (а) у пользователя может быть несколько устройств, теоретически на разных серверах; (б) при **миграции устройства** между серверами меняется именно `user_devices.server_id`, а коммерческая `subscriptions.server_id` может остаться/мигрировать отдельно; (в) координатор считает срез allowlist именно по `user_devices.server_id` — это его рабочий ключ.

Правило целостности: для активного устройства в multi-server-режиме `user_devices.server_id` **обязан** быть заполнен; FK на `servers.id` — мягкий на переходный период (4.7).

**Поток выдачи (multi-server):**
```
create/extend → IssuancePolicy.pick(server_id) → subscriptions.server_id = X
                                                → user_devices.server_id  = X (для устройств клиента)
                                                → Coordinator.on_change(user, server_id=X)
                                                → пересчёт allow для X → push на сервер X
```

### 4.7 Миграция устройства между серверами

Нетривиальный сценарий — фиксируем явно:

```
1. Запись намерения: user_devices.server_id  X → Y   (одной транзакцией)
2. Coordinator.on_change(server_id=Y): пересчёт allow(Y) + push(Y)  → устройство РАЗРЕШЕНО на Y
3. Coordinator.on_change(server_id=X): пересчёт allow(X) + push(X)  → устройство УБРАНО с X
4. Health-poll подтверждает обе version; до подтверждения — устройство временно валидно на ОБОИХ
   (осознанный выбор: лучше кратко «разрешён на двух», чем «запрещён на обоих» = клиент без связи).
```

Порядок шагов 2→3 важен: сначала **добавить на новый**, потом **убрать со старого** (fail-safe для платящего клиента). Для блокировки — обратный порядок (сначала убрать). Это прямое следствие fail-open: окно «разрешён на старом и новом» безопаснее для UX, чем окно «нигде».

### 4.8 План перехода single → multi (реалистичный, по этапам)

1. **Этап 1 (фундамент, без изменения поведения):** таблицы `servers` + `server_id` (nullable) в `subscriptions`/`user_devices`; `ServerRegistry`; `Coordinator` + `LocalFilePusher` (no-op для одного сервера); атомарная запись и version в payload (4.5). Поведение идентично текущему.
2. **Этап 2 (доставка):** `ServerAuthzPusher` вариант A (rsync+trigger); health-poll (4.4); feature-flag `multi_server_authz`; backfill-скрипт `server_id` для существующих записей; метрики push success/failure.
3. **Этап 2→3 (cutover):** второй сервер в прод; включить флаг; наблюдать «device allowed on unexpected server». Откат — флагом к single-server.
4. **Этап 3 (зрелость):** вариант B (signed HTTP push, version-reject); миграция устройств (4.7); опционально вариант C.

## 5. Риски внедрения и митигации

> Раздел добавлен по ревью. Это не теория — каждый риск вытекает из реального кода/модели и должен попасть в операционные процедуры и `OlcPanel_08`.

### 5.1 Race condition / частичная запись `authz.json` (критично)
- **Суть:** `Gate` читает по mtime-кэшу. Неатомарная запись = либо потерянные allow (клиенты отваливаются), либо неполный/битый срез. С прежним fail-open это означало **«пустим всех бесплатно»**; с принятым `last-known-good` (5.2) битый файл больше не открывает всех — `Gate` удержит последний валидный срез. **Но риск не снят полностью:** неатомарная запись *валидного-но-неполного* JSON может «зафиксироваться» как новое хорошее состояние (потеря части allow → отвал платящих), а частая запись в пределах гранулярности mtime — потеряться (4.5 п.3).
- **Митигация:** только атомарная запись (`tmp`+`fsync`+`rename`); никаких «дописываний на месте»; строгая валидация схемы и **монотонность version** в `Gate` (откат версии не обновляет LKG); вариант B подтверждает запись 200-ответом; CI-тест на оба инварианта: «битый/пропавший файл в режиме `lkg` НЕ расширяет множество allow» и «неполный валидный файл не теряет allow при атомарной записи» (см. 5.2).

### 5.2 Fail-open vs монетизация — РЕШЕНО: last-known-good кэш в `Gate` (вариант «б») ✅
- **Суть проблемы:** для *доступности* fail-open верен (сбой записи/чтения файла → клиенты не теряют связь). Для *платного* продукта тот же fail-open — **прямая дыра выручки и безопасности**: стоит файлу `authz.json` исчезнуть, обнулиться или побиться (частичная запись, ошибка диска, неудачный push, баг writer'а), как `Gate.Allowed()` начинает возвращать `true` для **любого** `deviceId` — включая неоплативших, истёкших и вручную заблокированных. То есть **самый частый класс отказов оборачивается тихой бесплатной раздачей сервиса**, причём без единого сигнала «что-то не так» на стороне клиента.

- **Рассмотренные варианты:**
  - **(а) Оставить fail-open as is.** Максимальный аптайм, нулевая доработка сервера. Но монетизация держится исключительно на внешнем мониторинге «файл свежий и валидный»; любая дыра в мониторинге = дыра в выручке. Отвергнут: перекладывает гарантию платности на операционные процедуры, а не на код.
  - **(в) Fail-closed после grace-периода.** При пропаже/порче файла `Gate` какое-то время работает по старым данным, затем **закрывает всех** (`Allowed()→false`). Надёжно против бесплатной раздачи, но при затяжном сбое панели/доставки **рубит и легитимных платящих клиентов** — прямой удар по тем, кто заплатил, из-за нашей же инфраструктурной проблемы. Отвергнут как основной режим (оставлен как опциональный «жёсткий потолок», см. ниже).
  - **(б) Last-known-good (LKG) кэш в `Gate` — ВЫБРАНО.** `Gate` держит в памяти **последний успешно загруженный и провалидированный** allowlist. При любой ошибке свежего чтения/парсинга (файл пропал, не парсится, не проходит схему/версионную проверку) `Gate` **не открывается всем и не закрывает всех — он продолжает применять предыдущий валидный срез**. Это «заморозка последнего известного хорошего состояния», а не «распахнуть двери».

- **Решение владельца (зафиксировано 14.06.2026):** в **paid-режиме** `Gate` работает по схеме **last-known-good (вариант «б»)**. Fail-open перестаёт быть поведением по умолчанию для продакшена; он сохраняется только как явный диагностический режим (`authz.fail_mode: open`) для отладки и для бесплатных/тестовых инсталляций.

- **Обоснование выбора (почему именно «б»):**
  1. **Закрывает дыру выручки, не наказывая платящих.** Самый вероятный инцидент — кратковременная порча/пропажа файла при записи или доставке. LKG превращает его в *no-op*: пока не приедет новый валидный файл, действует последний корректный allowlist. Неоплатившие не попадают «по случайности», платящие не теряют доступ.
  2. **Совпадает с уже выбранной философией файлового подхода (4.3).** Мы сознательно сделали серверы автономными: «упала панель — серверы живут по последнему `authz.json`». LKG — логическое завершение той же идеи на уровень ниже: «побился файл — `Gate` живёт по последнему валидному состоянию в памяти». Один и тот же принцип «деградируй до последнего хорошего, не до катастрофы» на обоих уровнях.
  3. **Дешевле и безопаснее grace-таймера (в).** Вариант «в» требует калибровки grace-периода (слишком короткий — рубит платящих при обычном сбое доставки; слишком длинный — превращается в тот же fail-open). LKG не требует подбора таймаута для базового случая: «держим последнее хорошее» — корректно и при секундном, и при многочасовом сбое.

- **Как работает LKG-кэш в `Gate` (механика):**
  ```
  on Allowed(deviceID):
    st, err := os.Stat(path)
    if err != nil (файл пропал):
        → ВЕРНУТЬ решение по lkgAllow (НЕ fail-open)
    if st.mtime == cached.mtime:
        → решение по текущему cached (как сейчас)
    else (mtime сдвинулся — пробуем перечитать):
        data, perr := parseAndValidate(path)   // JSON + схема + version-моноотонность
        if perr != nil (битый/неполный/откат версии):
            → НЕ трогаем cached, ВЕРНУТЬ решение по lkgAllow, инкремент authz_load_errors, alert
        else:
            cached   = data
            lkgAllow = data            // обновляем «последнее хорошее» ТОЛЬКО при успехе
            lkgValidAt = now
            → решение по cached
  ```
  Ключевые инварианты: (1) `lkgAllow` обновляется **исключительно** после успешной валидации; (2) ошибка чтения/парсинга **никогда** не очищает `lkgAllow` и **никогда** не открывает всех; (3) каждая ошибка инкрементит счётчик и порождает алерт (для 5.6/3.6), чтобы «тихая заморозка» не стала «тихим устареванием».

- **Требования, которые LKG накладывает на компоненты:**
  - **`Gate` (`internal/authz/authz.go`):** добавить поля `lkgAllow/lkgDeny`, `lkgValidAt`, `loadErrors`; новый конфиг `authz.fail_mode: lkg|open|closed` (по умолчанию в проде `lkg`); параметр `authz.lkg_max_age` — опциональный «жёсткий потолок»: если последнее валидное состояние старше N (напр. 24 ч) и свежих успешных загрузок нет, эскалировать (закрыть всех / алерт уровня critical) — это и есть встроенный вариант «в» поверх «б», но как осознанный край, а не основной режим. **Холодный старт:** если при запуске сервера файла нет вовсе (нет ни одного «хорошего» состояния) — поведение определяется `fail_mode` для cold-start: рекомендуется `closed` (пустой allow) с громким алертом, т.к. «никогда не видели валидного allowlist» ≠ «временно потеряли свежий».
  - **`authz_writer.write_authz_file(db)`:** атомарность (`tmp`+`fsync`+`os.rename`) становится **обязательной** (раньше — желательной): LKG защищает от *уже случившейся* порчи, но не должен её провоцировать. Writer обязан гарантировать монотонный `version`/`updated_at` (см. 4.5 п.2–3), чтобы validate-функция `Gate` могла отвергать откаты, не сбрасывая LKG.
  - **Процесс записи/доставки (4.3, A/B):** в варианте B endpoint `/internal/authz/update` валидирует payload **до** записи и отвечает `200` только после успешной атомарной записи — то есть «плохой» payload вообще не доходит до файла, и LKG даже не активируется. LKG — вторая линия обороны на случай, когда первая (валидация на доставке) пропустила порчу (например, повреждение на диске после записи).

- **Влияние на архитектуру (включая multi-server):**
  - **Семантика «свежести» становится наблюдаемой величиной.** Раз `Gate` может работать по устаревшему-но-валидному состоянию, health-poll (4.4/3.6) обязан отдавать не только «жив ли сервер», но и `applied_version`, `lkg_valid_at`, `load_errors`. Дашборд показывает «сервер X применяет authz возрастом 2 ч» — это новый, важный класс состояния, которого при fail-open не было (там было бинарно «открыт/по файлу»).
  - **Split-brain (5.3) смягчается, но не исчезает.** LKG гарантирует, что сбой доставки не «распахнёт» сервер; он застынет на последнем хорошем срезе. Это лучше прежнего fail-open (раньше застрявший сервер при порче файла пускал всех), но рассогласование «сервер X на version=10, сервер Y на version=12» остаётся — лечится теми же health-poll + алертами.
  - **Безопасность (08).** LKG закрывает основной сценарий «потеря файла = бесплатный VPN». Остаточный риск смещается на «устаревший allowlist»: заблокированный клиент может оставаться доступным ровно до тех пор, пока не приедет следующий валидный файл; это и ограничивает `lkg_max_age` + алерты.
  - **CI/тесты (5.1):** инвариант теперь формулируется жёстко и тестируемо: «битый/пропавший файл в режиме `lkg` НЕ расширяет множество разрешённых `deviceId` сверх последнего валидного». Это обязательный регрессионный тест сервера.

- **Компромиссы, которые мы принимаем сознательно:**
  - **Устаревание вместо распахивания.** Платим тем, что при длительном сбое заблокированный клиент может «жить» дольше положенного (по старому allowlist). Это управляемо (`lkg_max_age`, алерты) и несравнимо безопаснее бесплатной раздачи всем.
  - **Состояние в памяти `Gate`.** LKG живёт в RAM процесса. Перезапуск `olcrtc` при отсутствующем/битом файле = потеря LKG → отрабатывает cold-start policy (`closed`+alert). Поэтому рестарты сервера должны идти после проверки валидности файла (войдёт в `OlcPanel_07` как операционная процедура).
  - **Чуть больше кода в `Gate`.** Добавляем валидацию, версионную проверку и кэш-поля — но это локальное, хорошо тестируемое изменение одного пакета `internal/authz`, не затрагивающее транспорт/комнату/ключ.

> Итог: вопрос **закрыт**. В проде `authz.fail_mode = lkg`. Деталь дизайна `Gate`, writer'а и health-полей переносится в `OlcPanel_03` (сервис `Gate`/authz), `OlcPanel_07` (операционные процедуры рестарта/доставки) и `OlcPanel_08` (модель угроз: «потеря файла» больше не = «открытый доступ»).

### 5.3 Split-brain / окно несогласованности (multi-server)
- **Суть:** между push'ами устройство может быть разрешено на одном сервере и запрещено на другом; при потере связи панель↔сервер allowlist «застывает».
- **Митигация:** health-poll сравнивает применённую `version`/`updated_at` с ожидаемой → дашборд (3.6) + алерт в Telegram (2.4); метрика «device allowed on unexpected server»; порядок шагов миграции/блокировки из 4.7; best-effort + ручной re-push из админки. Полностью устранить нельзя — **принять и мониторить** (как и в `olcrtc-panel-architecture.md`).

### 5.4 Центральная панель как SPOF авторизации
- **Суть:** раньше авторизация была локальной на каждом сервере; central panel централизует риск. Деградация панели бьёт по всему флоту (особенно вариант C).
- **Митигация:** файловый подход (A/B) специально оставляет серверы автономными при падении панели; бэкап БД (3.2); усиленная безопасность входа (3.1) поднимается в приоритете на multi-server.

### 5.5 Backfill и недооценка ручной работы
- **Суть:** перевод существующих `user_devices`/`subscriptions` на `server_id` — недооцениваемый объём; ошибки backfill = клиенты на «не своём» сервере.
- **Митигация:** идемпотентный backfill-скрипт + dry-run + отчёт расхождений; dual-write период; feature-flag для отката.

### 5.6 Технический долг наблюдаемости
- **Суть:** без логов/метрик по операциям координатора отладка multi-server невозможна.
- **Митигация:** аудит (2.2) пишет каждый push/результат; метрики push success/failure per server; это **предусловие** Этапа 2, а не «потом».

## 8. С чего начать (рекомендация)

**Спринт «монетизация + фундамент» (Этап 1):**
1. **Авто-блок по сроку** (Cron→`authz_writer`) — #1.
2. **Фундамент координации** — #2: таблицы `servers`/`server_id`, `Coordinator`+`LocalFilePusher` (no-op), **атомарная запись + version в payload** (4.5). Меняет схему, не меняет поведение.
3. **Аудит** (#3) и **Swagger** (#4).

Почему именно так (исправление противоречия из ревью): «монетизация» и «фундамент масштабирования» — **один спринт**, потому что авто-блок уже пишет `authz.json`, и закладывать его без атомарности/версии/`server_id` — значит переписывать через месяц. Реальный мульти-push при этом откладываем — включаем флагом на Этапе 2.

**Спринт «операционка» (Этап 2):** event-шина → Telegram (#6, вкл. push-failure) + напоминания (#5) + бэкап (#8) + безопасность входа (#7); затем `ServerAuthzPusher` вариант A + health-poll (#9) и шаблоны подписки (#10).

**Спринт «масштабирование» (Этап 2→3):** мониторинг флота (#11), вариант B push (#13), миграция устройств (4.7). Webhook (#12) — когда дойдём до автоплатежей.

**Решено (14.06.2026):** `Gate` в paid-режиме работает по **last-known-good** (`authz.fail_mode: lkg`, вариант «б», см. 5.2). Это влияет на дизайн writer'а (обязательная атомарность + монотонный version), на `Gate` (поля LKG, валидация, `lkg_max_age`, cold-start policy) и на health-поля (`applied_version`/`lkg_valid_at`/`load_errors`). Детали — в `OlcPanel_03/07/08`.

**Чего сознательно НЕ делаем:** ничего из раздела 6.

## 9. Влияние на архитектурные документы (для итерации)

При утверждении правки лягут в:

- `OlcPanel_02_Модель_данных` — `server_id` в `subscriptions` **и** `user_devices` (семантика 4.6); event/notification-таблицы; статусы blocked-by-expiry; `servers.api_base`/health-поля; поля версии applied-authz.
- `OlcPanel_03_Архитектура_сервисов` — `SchedulerService` (Cron), `NotificationService` (Telegram/Webhook), `ServerRegistry`, `IssuancePolicy`, `MultiServerAuthzCoordinator`, `ServerAuthzPusher` (A/B), `ServerHealthService`.
- `OlcPanel_04_API` — CRUD серверов, `/internal/authz/update` (вариант B), health-эндпоинты, Swagger.
- `OlcPanel_05_Бизнес_логика` — алгоритм авто-блока/напоминаний; выбор сервера; пересчёт allow по `server_id`; порядок шагов миграции/блокировки (4.7).
- `OlcPanel_07_Инфраструктура` — health-эндпоинты и trigger на olcrtc-серверах; rsync/HTTPS доставка; fleet-скрипты; бэкап-крон.
- `OlcPanel_08_Безопасность` — **принятое решение last-known-good (5.2)**: модель угроз «потеря/порча `authz.json` ≠ открытый доступ», остаточный риск «устаревший allowlist» + `lkg_max_age`/алерты, cold-start policy `closed`; 2FA/секретный путь/fail2ban; подпись/`mTLS` для push; SPOF-митигации.
