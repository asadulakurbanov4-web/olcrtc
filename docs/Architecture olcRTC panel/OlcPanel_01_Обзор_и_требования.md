# OlcPanel — Раздел 1: Обзор системы и требования
## Бизнес-сценарии · Роли · Функциональные требования · Нефункциональные требования · User Stories

> **Версия:** 2.0 (переписано под модель `deviceId` + общая комната + Phase 1; soft-block через `authz.json`; трафик де-приоритизирован; добавлены требования флота и LKG-надёжности)
> **Статус:** 🟡 В разработке
> **Зависимости:** [[OlcPanel_00_Введение]] — прочитать перед этим разделом (Решения 1–5)

> ⚠️ **Что изменилось против v1.x.** Раньше монетизация и изоляция строились на «уникальный ключ + комната на клиента» и пер-клиентском трафик-аккаунтинге. В v2.0: **одна общая комната + один ключ**, различие клиентов по `deviceId`, блокировка — **soft-block через `authz.json`/`Gate`** (а не удаление правил iptables), продажа — **по сроку**, не по объёму. Требования ниже переписаны под это.

## 1.1 Обзор бизнеса

### 1.1.1 Бизнес-модель

OlcPanel обслуживает **микробизнес по продаже VPN-доступа** — «частный VPN-провайдер для знакомых»: распространённая в РФ модель монетизации технической экспертизы в условиях DPI/ТСПУ.

```
┌──────────────────────────────────────────────────────────────────┐
│                        БИЗНЕС-МОДЕЛЬ                              │
├──────────────────────────────────────────────────────────────────┤
│  Клиент                                                           │
│  • Хочет доступ к заблокированным сервисам                        │
│  • Доверяет знакомому больше, чем анонимному VPN                  │
│  • Платит ежемесячно (300–500 ₽); техграмотность низкая           │
│  • Получает ссылку подписки → вставляет в OlcBox → работает       │
│                                                                   │
│  Администратор (владелец)                                         │
│  • Имеет VPS с OlcRTC-сервером (позже — флот VPS)                 │
│  • Продаёт доступ через личные контакты; оплата на карту/крипту   │
│  • Создаёт клиента в панели, выдаёт ссылку подписки               │
│  • Панель сама блокирует по истечении срока (soft-block)          │
│                                                                   │
│  Монетизация (продажа ПО СРОКУ)                                   │
│  10 клиентов × 400 ₽/мес = 4 000 ₽/мес                           │
│  20 клиентов × 400 ₽/мес = 8 000 ₽/мес                           │
│  VPS (Европа): ~800–1 500 ₽/мес; маржа при 20 клиентах ~6 500 ₽  │
└──────────────────────────────────────────────────────────────────┘
```

> Продажа по сроку, не по объёму — прямое следствие Решения 5 ([[OlcPanel_00_Введение]]): WebRTC-туннель не даёт пер-клиентскую статистику трафика, поэтому лимиты по ГБ в v2.0 не продаются.

### 1.1.2 Технологическая основа и что решает панель

Продукт построен на **OlcRTC** — обфускированном туннеле поверх WebRTC/Jitsi: трафик неотличим от видеозвонка, ТСПУ не шейпит; клиент ставит одно приложение (OlcBox), конфиг — ссылка подписки.

**Проблемы без панели → решение OlcPanel (v2.0):**

| Проблема без панели | Решение с OlcPanel |
|---|---|
| Клиенты в Excel/Telegram → легко потерять | Централизованная БД (SQLite, источник истины) |
| Ручное добавление `deviceId` в `authz.json` | Авто-запись `authz.json` через `authz_writer` при любом изменении доступа |
| Забыл отключить неоплатившего | Авто-блок по сроку (Cron → `authz.json`) |
| «Отключил», а клиент всё ещё ходит | `Gate` перечитывает `authz.json` и режет живые сессии (`EnforceInterval`) |
| Сбой файла раздаёт VPN всем бесплатно | `Gate` в режиме **last-known-good** (Решение 4): сбой ≠ открытый доступ |
| Рост → один сервер не тянет | Фундамент флота с Этапа 1: `server_id`, координатор, доставка среза |

> Заметьте: «чей ключ отозвать при утечке» больше не стоит как в v1.x — ключ общий. Утечка ссылки подписки решается на уровне `deviceId` (отвязка устройства), а не сменой `crypto.key`. См. UC-04.

## 1.2 Роли в системе

### 1.2.1 Актёры

В MVP — **одна роль администратора**. Мульти-админ/реселлеры — Этап 3 ([[OlcPanel_Анализ_Marzban_3xui]], 3.8).

```
┌─────────────────────────────────────────────────────────────────┐
│                           РОЛИ v2.0                              │
├─────────────────────────────────────────────────────────────────┤
│  👤 АДМИНИСТРАТОР (единственная роль в MVP)                       │
│     Кто: владелец VPN-бизнеса                                    │
│     Может: CRUD клиентов/подписок/устройств; выдача ссылок;       │
│            ручной блок/разблок; управление тарифами; (флот)       │
│            реестр серверов и контроль применения authz           │
│                                                                  │
│  🤖 СИСТЕМА (APScheduler + координатор authz)                     │
│     • Авто-блок по истечении подписки (→ authz.json)             │
│     • Напоминания об истечении за N дней                         │
│     • (флот) пересчёт среза allow по server_id + доставка        │
│     • (флот) health-poll: applied_version / lkg_valid_at         │
│     • Очистка логов; авто-бэкап БД                               │
│                                                                  │
│  📱 КЛИЕНТ VPN (внешний актёр, без доступа к панели)             │
│     • Взаимодействует только через OlcBox                        │
│     • Шлёт deviceId (x-hwid) при загрузке подписки и в handshake │
│     • Получает ссылку подписки (olcrtc:// URI), не сам ключ      │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2.2 Будущие роли

| Роль | Этап | Описание |
|---|---|---|
| **Client (self-service)** | Этап 3 | Личный кабинет/бот: статус подписки, свои устройства, ротация |
| **Reseller** | Этап 3 | Ограниченный пул клиентов (RBAC) |
| **SuperAdmin / Fleet** | Этап 2–3 | Управление флотом серверов из центральной панели |

## 1.3 Бизнес-сценарии (Use Cases)

### UC-01: Подключение нового клиента (v2.0)

**Актёр:** Администратор · **Триггер:** знакомый оплатил доступ

```
1. Договорились о тарифе/сроке.
2. Панель → «Клиенты» → «+ Новый клиент»: имя, контакт, тариф (срок), заметка,
   (флот) сервер по умолчанию через IssuancePolicy → server_id.
3. «Создать». Система:
   - создаёт client + subscription (start=сегодня, expires_at=+срок);
   - НЕ генерирует пер-клиентский ключ/комнату — они общие из server.yaml;
   - формирует ссылку подписки olcrtc:// (olcbox получит общую комнату+ключ);
   - помечает: устройство привяжется при первой загрузке подписки.
4. Админ отправляет клиенту ССЫЛКУ ПОДПИСКИ.
5. Клиент открывает OlcBox → вставляет ссылку → при загрузке подписки olcbox
   шлёт x-hwid → панель ловит его в /api/sub/{token} → bind_device →
   authz_writer.write_authz_file(db) → authz.json содержит deviceId →
   (флот) Coordinator.on_change → push на server_id.
6. Gate.Allowed(deviceID)=true → клиент подключается.

Постусловие: клиент активен; deviceId в allowlist; запись в audit_log.
```

**Acceptance Criteria:**
- [ ] Создание клиента ≤ 30 c; ссылка подписки формируется мгновенно.
- [ ] Ссылка корректно открывается в OlcBox и отдаёт общую комнату+ключ из `server.yaml` (не пер-клиентские).
- [ ] При первой загрузке подписки `deviceId` появляется в `authz.json` (атомарно, с возросшим `version`).
- [ ] Клиент появляется в списке как «Активен»; устройство — в списке устройств клиента.

### UC-02: Продление подписки

```
1. Получена оплата.
2. Панель → клиент → «Продлить»: +30/+60/+90 дней или дата.
   Новая дата считается от ТЕКУЩЕЙ expires_at (не от сегодня), если подписка ещё активна.
3. Система: обновляет expires_at; статус → active; если был soft-block —
   возвращает устройства клиента в allow → authz_writer → (флот) push;
   пишет запись в историю платежей и audit_log.

Постусловие: подписка продлена, deviceId(ы) снова в allowlist.
```

**Acceptance Criteria:**
- [ ] «Продлить» доступна для активного/истёкшего/заблокированного клиента.
- [ ] Продление истёкшего автоматически возвращает устройства в `authz.json`.
- [ ] История платежей сохраняется; новая дата корректна в списке.

### UC-03: Блокировка клиента (soft-block через authz.json)

**Актёр:** Система (авто) или Администратор (вручную) · **Триггер:** `expires_at < now()` либо ручной блок

```
Автоматический путь:
1. APScheduler сканирует subscriptions.expires_at.
2. Для просроченных: subscription→expired, user_devices.is_active=false.
3. authz_writer.write_authz_file(db): deviceId(ы) клиента УБИРАЮТСЯ из allow
   (атомарно, version++). Запись/удаление — НЕ delete клиента (история нужна).
4. (флот) Coordinator.on_change(server_id) → пересчёт allow → push.
5. Gate перечитывает authz.json (mtime) и в пределах EnforceInterval
   рвёт живые сессии заблокированного deviceId (removePeerSession).

Ручной путь: карточка клиента → «Блок» → подтверждение → те же шаги.

Постусловие: deviceId вне allowlist; активная сессия рвётся ≤ EnforceInterval;
клиент сохранён в БД (soft-block, не delete).
```

**Acceptance Criteria:**
- [ ] Авто-блок срабатывает ≤ 1 ч после истечения; ручной — немедленно пишет `authz.json`.
- [ ] Заблокированный клиент остаётся в БД со статусом «Заблокирован/Истёк».
- [ ] Активная сессия рвётся в пределах `EnforceInterval` (а не «до следующего реконнекта»).
- [ ] **Сбой записи `authz.json` не открывает доступ** (LKG, Решение 4) — регрессионный тест.

### UC-04: Утечка ссылки подписки / отвязка устройства

**Триггер:** клиент поделился ссылкой/устройством с посторонним

```
В v2.0 ключ ОБЩИЙ, поэтому «сменить crypto.key» не применяется. Рычаги:
1. Отвязать лишний deviceId: карточка клиента → «Устройства» → удалить →
   authz_writer убирает этот deviceId из allow → push. Чужое устройство отваливается.
2. Ротация sub-токена клиента (olcrtc_key): старая ссылка подписки инвалидируется
   (/api/sub/{old} → 404), при следующей загрузке olcbox получает новую.
3. (Этап 2+) лимит числа устройств на клиента: лишние не биндятся.

Постусловие: посторонний deviceId не в allowlist; легитимные устройства работают.
```

**Acceptance Criteria:**
- [ ] Удаление устройства немедленно убирает его `deviceId` из `authz.json`.
- [ ] Ротация sub-токена рвёт старую ссылку подписки, не трогая активные устройства.
- [ ] Остальные клиенты (общий ключ) **не** затронуты — блокировка точечная по `deviceId`.

### UC-05: Дашборд (ежедневный мониторинг)

```
Виджеты: всего клиентов; активных подписок; истекает ≤ N дней; истёкших;
(флот) статус серверов: жив? applied_version свежий? lkg_valid_at не устарел?
load_errors=0? Блок «Требуют внимания» (истекающие + истёкшие + серверы с
устаревшим authz). Статус OlcRTC-сервера(ов).
```

**Acceptance Criteria:**
- [ ] Дашборд < 500 мс; данные не старше 5 мин.
- [ ] «Требуют внимания» отсортирован по срочности; включает серверы с устаревшим `applied_version`/`load_errors>0`.

### UC-06: Карточка клиента

```
Основное: имя, контакт, дата создания, тариф; подписка (start/expires/осталось);
устройства (deviceId, last_used_at, server_id, активно/нет); история подписок/
платежей; ссылка подписки (URI + QR); журнал действий по клиенту.
```

### UC-07: Тарифные планы

```
Список тарифов (название, срок, цена; лимит трафика как «спящее» поле —
учёт не настроен). Редактирование тарифа применяется к НОВЫМ подпискам.
```

### UC-08: Восстановление после перезагрузки сервера (с учётом LKG)

```
1. systemd поднимает olcrtc.service и olcpanel.service.
2. Gate при старте читает authz.json:
   - файл валиден → загрузка allow, lkgValidAt=now;
   - файл отсутствует/битый при ХОЛОДНОМ старте (нет LKG в памяти) →
     cold-start policy = closed + громкий алерт (НЕ open). См. Решение 4.
   Поэтому рестарт olcrtc должен идти ПОСЛЕ проверки валидности authz.json
   (операционная процедура — [[OlcPanel_07_Инфраструктура]]).
3. Клиенты переподключаются (olcbox reconnect).
Действий администратора: 0 (при валидном файле).
```

### UC-09: «Не работает VPN» — диагностика

```
1. Карточка клиента: подписка не истекла? устройство в allow?
2. Сервер клиента (server_id): жив? applied_version совпадает с ожидаемым?
   load_errors=0? (если Gate на LKG из-за битого файла — это видно на дашборде).
3. Последнее подключение (last_used_at). При проблеме сервера — рестарт/ре-push authz.
```

### UC-10: Смена несущего Jitsi (комната заблокирована)

```
В v2.0 комната ОБЩАЯ, поэтому смена проще, чем в v1.x (не нужно обновлять
пер-клиентские URI):
1. Настройки → профиль(и) сервера в server.yaml (failover уже на несколько несущих).
2. olcbox получает несущие из подписки/конфига; при наличии failover-профилей
   переключение происходит без ручной рассылки каждому.
3. Массовая рассылка нужна только если меняется сам формат ссылки подписки.
```

## 1.4 Функциональные требования

### 1.4.1 Аутентификация (AUTH)

| ID | Требование | Приоритет |
|---|---|---|
| AUTH-01 | Логин/пароль для доступа к панели | P0 |
| AUTH-02 | Пароли — bcrypt (cost ≥ 12), хранятся в `panel.env`, не в коде | P0 |
| AUTH-03 | JWT с expire (24 ч) + `fp`-инвалидатор (смена пароля рвёт старые токены без рестарта) | P0 |
| AUTH-04 | Смена пароля из настроек, атомарная запись в `panel.env` | P0 |
| AUTH-05 | Логирование всех попыток входа (IP, время) | P1 |
| AUTH-06 | fail2ban / лимит попыток входа | P1 (на multi-server — P0) |
| AUTH-07 | 2FA (TOTP) + секретный URL-путь панели | P2 (на multi-server раньше) |

> На multi-server центральная панель — SPOF авторизации (5.4 в [[OlcPanel_Анализ_Marzban_3xui]]), поэтому AUTH-06/07 поднимаются в приоритете. Детали — [[OlcPanel_08_Безопасность]].

### 1.4.2 Клиенты (CLIENTS)

| ID | Требование | Приоритет |
|---|---|---|
| CL-01 | Создание клиента: имя, контакт, заметка, тариф, (флот) server_id | P0 |
| CL-02 | Список с фильтром по статусу и поиском | P0 |
| CL-03 | Детальная карточка (вкл. устройства и историю) | P0 |
| CL-04 | Редактирование (имя/контакт/заметка) | P0 |
| CL-05 | Soft-delete (история сохраняется); удаление убирает deviceId из allow | P1 |
| CL-06 | Статусы: Активен / Истекает / Истёк / Заблокирован / Удалён | P0 |

### 1.4.3 Доступ и устройства (DEVICES / SUBSCRIPTION)

| ID | Требование | Приоритет |
|---|---|---|
| DV-01 | Ссылка подписки `olcrtc://…` отдаёт ОБЩУЮ комнату+ключ из `server.yaml` | P0 |
| DV-02 | Загрузка подписки `/api/sub/{token}` ловит `x-hwid` → `bind_device` | P0 |
| DV-03 | Список устройств клиента: `deviceId`, `last_used_at`, `server_id`, активно/нет | P0 |
| DV-04 | Удаление/деактивация устройства → убрать `deviceId` из `authz.json` | P0 |
| DV-05 | Ротация sub-токена (`olcrtc_key`) — старая ссылка → 404 | P1 |
| DV-06 | (Этап 2) лимит числа устройств на клиента | P2 |
| DV-07 | QR ссылки подписки | P1 |

### 1.4.4 Подписки (SUBSCRIPTIONS)

| ID | Требование | Приоритет |
|---|---|---|
| SUB-01 | Подписка создаётся вместе с клиентом согласно тарифу | P0 |
| SUB-02 | Продление: +30/+60/+90 или дата (от текущей `expires_at`) | P0 |
| SUB-03 | Ручной soft-block / разблок (→ `authz.json`) | P0 |
| SUB-04 | Авто-блок по сроку (APScheduler, ≤ 1 ч) | P0 |
| SUB-05 | «Требуют внимания»: истекающие + истёкшие | P0 |
| SUB-06 | История платежей (дата, сумма, период, способ) | P1 |
| SUB-07 | Предупреждение об истечении за `warn_days_before` | P1 |

### 1.4.5 Тарифы (PLANS)

| ID | Требование | Приоритет |
|---|---|---|
| PLN-01 | Предустановленные тарифы по сроку (Trial/Базовый/Стандарт/Безлимит-по-сроку) | P0 |
| PLN-02 | Редактирование (не затрагивает активные подписки) | P1 |
| PLN-03 | Поле «лимит трафика» присутствует, но помечено «учёт не настроен» (спящее) | P1 |

### 1.4.6 Контракт authz (AUTHZ) — ядро Phase 1

| ID | Требование | Приоритет |
|---|---|---|
| AZ-01 | `authz_writer` пишет `authz.json` `{version, updated_at, allow[], deny[]}` **атомарно** (`tmp`+`fsync`+`rename`) | P0 |
| AZ-02 | `version`/`updated_at` **монотонно возрастают** (для version-reject на сервере) | P0 |
| AZ-03 | Запись триггерится: create/extend/block/unblock/delete, sub-fetch, scheduler | P0 |
| AZ-04 | `Gate` в проде — `fail_mode: lkg` (Решение 4); `open` только для диагностики | P0 |
| AZ-05 | `Gate` валидирует схему и отвергает откат `version`, не сбрасывая LKG | P0 |
| AZ-06 | `lkg_max_age` + алерт при устаревании; cold-start policy = `closed`+alert | P1 |
| AZ-07 | Регрессионный тест: «битый/пропавший файл в `lkg` НЕ расширяет allow» | P0 |

### 1.4.7 Флот серверов (FLEET) — фундамент с Этапа 1

| ID | Требование | Приоритет |
|---|---|---|
| FL-01 | Таблица `servers` (server_id, статус, api_base, priority) | P0 (скелет) |
| FL-02 | `server_id` (nullable) в `subscriptions` и `user_devices` | P0 (скелет) |
| FL-03 | `Coordinator` + `LocalFilePusher` (no-op для одного сервера) | P0 (скелет) |
| FL-04 | `ServerAuthzPusher` вариант A (rsync+trigger) | P1 (Этап 2) |
| FL-05 | `ServerHealthService`: poll `/health` (applied_version, lkg_valid_at, load_errors) | P1 (Этап 2) |
| FL-06 | `/internal/authz/update` (signed/mTLS) с version-reject — вариант B | P2 (Этап 3) |

### 1.4.8 Дашборд и настройки (DASHBOARD / SETTINGS)

| ID | Требование | Приоритет |
|---|---|---|
| DSH-01 | Виджеты клиентов/подписок/истекающих | P0 |
| DSH-02 | Статус сервера(ов); (флот) свежесть applied authz | P0 |
| DSH-03 | «Требуют внимания» (вкл. серверы с устаревшим authz/load_errors) | P0 |
| SET-01 | Смена пароля; настройка `warn_days_before` | P0/P1 |
| SET-02 | (флот) CRUD серверов, ручной re-push authz | P1 |
| SET-03 | Просмотр логов / рестарт OlcRTC (с проверкой валидности `authz.json`) | P1 |

> Модуль TRAFFIC из v1.x **исключён из P0/P1**: пер-клиентский трафик-аккаунтинг не реализуем на WebRTC-туннеле (Решение 5). Поля `traffic_*` остаются «спящими» до появления серверной телеметрии (Этап 3, см. 3.7 аналитики).

## 1.5 Нефункциональные требования

### 1.5.1 Производительность

| Требование | Метрика |
|---|---|
| Загрузка страницы | < 500 мс |
| API CRUD (p95) | < 100 мс (SQLite) |
| Запись `authz.json` | атомарно, < 50 мс; без блокировки запросов панели |
| RAM FastAPI / Nginx | < 100 МБ / < 30 МБ |
| `Gate.Allowed()` | O(1) по кэшу allow; перечтение только при сдвиге mtime |

### 1.5.2 Надёжность (ключевой раздел v2.0)

| Требование | Метрика / правило |
|---|---|
| Uptime панели | ≥ 99% |
| Uptime OlcRTC | ≥ 99.5% (systemd Restart=always) |
| **Сбой `authz.json` ≠ открытый доступ** | В `fail_mode: lkg` битый/пропавший файл удерживает последний валидный allow (Решение 4). Тест AZ-07. |
| **Автономность сервера при падении панели** | `Gate` продолжает по последнему `authz.json`; панель не на критическом пути handshake |
| Окно применения блокировки | ≤ `EnforceInterval` для живых сессий |
| Потеря данных при перезагрузке | 0 (SQLite + fsync; атомарная запись authz) |
| Восстановление после перезагрузки VPS | < 60 c (при валидном `authz.json`; иначе cold-start `closed`+alert) |
| Бэкап БД (+ `authz.json`, `server.yaml`) | Ежедневно; ротация 7 дней |

### 1.5.3 Безопасность

| Требование | Реализация |
|---|---|
| Только HTTPS | Nginx + Let's Encrypt; HTTP→HTTPS |
| Авторизация на всех API кроме публичных | JWT middleware; публичные: `/api/sub/{token}`, `/api/auth/login` |
| Пароли | bcrypt cost≥12 в `panel.env`; `fp`-инвалидатор |
| `authz.json` целостность | Атомарная запись; монотонный `version`; права файла только для сервиса |
| Доставка authz (флот) | Вариант B: подпись/`mTLS`, version-reject, 200 только после записи |
| Защита входа | fail2ban (P1) → 2FA/секретный путь на multi-server |
| SPOF центральной панели | Файловый контракт оставляет серверы автономными; бэкап БД; усиленный вход |

### 1.5.4 Масштабируемость

| Этап | Клиентов / серверов | Изменения |
|---|---|---|
| MVP (Этап 1) | 1–50 / 1 | Текущая модель + скелет флота (no-op pusher) |
| Этап 2 | 50–150 / 2–5 | Pusher A (rsync), health-poll, feature-flag `multi_server_authz` |
| Этап 3 | 150–500 / 5+ | Pusher B (signed HTTP), миграция устройств, PostgreSQL при необходимости |

### 1.5.5 Сопровождаемость и наблюдаемость

| Требование | Реализация |
|---|---|
| Логирование | `logging` + journald (DEBUG/INFO/WARN/ERROR) |
| Аудит | Каждое действие и каждый push authz → `audit_log` (предусловие multi-server, 5.6) |
| Метрики | `authz_load_errors`, push success/failure per server, `lkg_valid_at` |
| Swagger | `/docs` (под авторизацией) |
| Миграции | Alembic, схема `servers`/`server_id`/health-поля закладывается на Этапе 1 |
| Структура | `routers/ models/ services/` (Phase 1 рефакторинг уже разнёс `main.py`) |

## 1.7 Ограничения и зависимости

### 1.7.1 Технические ограничения

| Ограничение | Влияние |
|---|---|
| RAM ~1.9 ГБ | FastAPI 1 воркер; SQLite (без PostgreSQL/Redis на старте) |
| Один транспорт (WebRTC/Jitsi) | Нет inbounds/протоколов/пер-клиентского трафика (Решение 5) |
| Общая комната + общий ключ | Изоляция/блокировка — только по `deviceId`, не по ключу |
| Go-сервер уже собран | Доработки `Gate` (LKG) — локально в `internal/authz`, аддитивно |

### 1.7.2 Зависимости от внешних систем

| Система | Риск | Митигация |
|---|---|---|
| Несущие Jitsi (meet1.arbitr.ru, conference.ct.placetime.team) | Высокий (блок/падение) | Несколько профилей + failover в `server.yaml` |
| Let's Encrypt | Низкий | Certbot auto-renew |
| OlcRTC binary + `Gate` | Высокий | systemd Restart=always; LKG; мониторинг свежести authz |
| Доставка `authz.json` (флот) | Средний | Атомарность, version-reject, health-poll, ре-push из админки |

### 1.7.3 Зависимости между модулями

```
AUTH ───────────────┐
                    ↓
PLANS ─→ SUBSCRIPTIONS ─→ CLIENTS ─→ DEVICES ─→ AUTHZ(authz_writer) ─→ Gate
                    │                                    │
                    ↓                                    ↓
              SCHEDULER (авто-блок)              FLEET(Coordinator/Pusher/Health)
                    │                                    │
                    └──────────────→ DASHBOARD ←─────────┘
```

- **AUTH** реализуется первым (блокирует всё).
- **AUTHZ (authz_writer + Gate)** — ядро монетизации; всё, что меняет доступ, проходит через него.
- **FLEET** — скелет с Этапа 1 (no-op), наполняется на Этапе 2.

## 1.8 Критерии готовности MVP (Этап 1)

### Must Have (P0)
- [ ] Авторизация (JWT + bcrypt + `fp`).
- [ ] Создание клиента + выдача ссылки подписки (общая комната+ключ).
- [ ] Привязка устройства по `x-hwid` при загрузке подписки.
- [ ] Список клиентов со статусами; карточка с устройствами.
- [ ] Продление; ручной soft-block/разблок.
- [ ] Авто-блок по сроку (≤ 1 ч).
- [ ] `authz_writer`: **атомарная запись + монотонный `version`**.
- [ ] `Gate` в режиме **`fail_mode: lkg`** + тест AZ-07 (сбой ≠ открытый доступ).
- [ ] Скелет флота: `servers`, `server_id`, `Coordinator`+`LocalFilePusher` (no-op).
- [ ] Дашборд (клиенты/подписки/статус сервера); HTTPS.

### Should Have (P1)
- [ ] QR ссылки подписки; удаление устройства; ротация sub-токена.
- [ ] Напоминания об истечении; история платежей.
- [ ] Аудит всех действий; `lkg_max_age` + алерт устаревания.
- [ ] Бэкап БД (+ `authz.json`, `server.yaml`).

### Nice to Have (P2 / Этап 2–3)
- [ ] Telegram-уведомления (вкл. push-failure); 2FA/секретный путь/fail2ban.
- [ ] Pusher A (rsync) + health-poll; мониторинг флота на дашборде.
- [ ] Лимит устройств на клиента; шаблоны страницы подписки.
