# OlcPanel — Архитектура системы управления VPN-клиентами
## Техническая спецификация и системное проектирование

> **Статус:** 🟡 В активной разработке
> **Версия документа:** 2.0 (переписано под актуальную модель: одна общая комната + `deviceId` + Phase 1; зафиксировано решение по `Gate` — last-known-good)
> **Дата обновления:** 14 июня 2026
> **Проект:** OlcPanel — панель управления подписками OlcRTC
> **Сервер:** Ubuntu 22.04 LTS, VPS (зарубежный) → горизонтальный scaling на флот дешёвых VPS
> **Стек:** Python 3.11 · FastAPI · SQLite · Vanilla JS · Nginx + интеграция Phase 1 (`authz_writer` + `Gate`)

> ⚠️ **Зачем версия 2.0.** Версия 1.x этого документа описывала раннюю модель «уникальная комната + уникальный ключ на каждого клиента» и пер-клиентский трафик-аккаунтинг как ядро. Эта модель **отвергнута** (см. 0.2): пер-клиентские комнаты дали нестабильность соединения («нет пинга»), а WebRTC-туннель не отдаёт пер-клиентскую статистику трафика. Актуальная модель — **одна общая комната + один общий ключ + блокировка по `deviceId` через `authz.json`/`Gate` (Phase 1)**. Документ 2.0 переписан под неё и согласован с аналитическим входом [[OlcPanel_Анализ_Marzban_3xui]].

## 0. Введение

### 0.1 Контекст и предпосылки

Система создаётся в условиях активного применения DPI (Deep Packet Inspection) российским регулятором — ТСПУ (Технические Средства Противодействия Угрозам). Классические VPN-протоколы (WireGuard, OpenVPN, VLESS+Reality) подвергаются шейпингу: TCP-сессии «замораживаются» после передачи 15–20 КБ, что делает прямое VPN-соединение нестабильным или невозможным.

В качестве обхода выбран **OlcRTC** — прокси-туннель поверх WebRTC (SCTP DataChannel). Для ТСПУ трафик выглядит как обычный видеозвонок через Jitsi Meet и не подпадает под шейпинг. Сам протокол решает только техническую задачу соединения. Чтобы превратить OlcRTC в коммерческий продукт (продажа доступа по подписке), нужна система управления, которая автоматизирует:

- управление жизненным циклом клиента: **создание → выдача доступа → продление → блокировка/отключение**;
- **автоматическую блокировку** по истечении подписки или неоплате;
- подготовку к **горизонтальному масштабированию** (флот серверов OlcRTC под одной центральной панелью);
- **минимизацию операционной нагрузки** на владельца сервиса.

Эту систему — **OlcPanel** — и описывает документ.

### 0.2 Ключевые архитектурные решения и их обоснование

> Это самый важный подраздел: он фиксирует, что в v2.0 изменилось по сравнению с v1.x и почему. Все остальные документы (02–08) исходят из этих решений.

#### Решение 1. Идентификация клиента через `deviceId`, а не через «комнату+ключ на клиента»

**Было (v1.x, отвергнуто):** каждому клиенту панель генерировала уникальную Jitsi-комнату и уникальный `crypto.key`. Изоляция строилась на уникальности ключа.

**Стало (v2.0):** **одна общая комната + один общий ключ шифрования** на сервер; клиенты различаются по `deviceId` (`x-hwid`), который `olcbox` шлёт при загрузке подписки и в handshake. Разрешение доступа — через файл `authz.json` (его пишет панель компонентом `authz_writer`) и серверный компонент `Gate` (`internal/authz/authz.go`, метод `Gate.Allowed(deviceID)`).

**Обоснование:**
- Модель «комната на клиента» на практике дала **нестабильность соединения** («нет пинга»): клиент заходил в свою пустую комнату, где сервера не было (сервер слушал свои фиксированные комнаты из `server.yaml`). Это и была корневая причина инцидента, с которого начался пересмотр архитектуры.
- Одна общая комната + один ключ **проще в эксплуатации и масштабировании**: сервер слушает заведомо «свои» комнаты, конфиг выдачи одинаков.
- Блокировка по `deviceId` отключает конкретного клиента **без влияния на остальных** в той же комнате — то есть монетизационный рычаг («отрубить неоплатившего») работает, несмотря на общий ключ.
- Модель уже **частично реализована и проверена** (Phase 1): панель пишет `authz.json`, сервер читает его через `Gate`.

**Вывод:** отказ от «одна комната — один клиент» в пользу `deviceId` + `authz.json` — осознанное, проверенное на инциденте решение. Детали схемы — [[OlcPanel_02_Модель_данных]], потоки — [[OlcPanel_03_Архитектура_сервисов]].

#### Решение 2. Разделение ответственности: панель = бизнес-логика, сервер = решение о доступе в реальном времени

**Решение:** панель отвечает за бизнес-логику (клиенты, подписки, устройства, выдача, продление, блокировка) и **материализует** результат в `authz.json`. Сервер (`Gate`) в реальном времени на handshake принимает решение «пускать/нет» по этому файлу. `authz.json` — **контракт** между панелью и сервером.

**Обоснование:**
- Снижает связанность: панель и сервер развиваются и тестируются независимо.
- **Сервер автономен при падении панели**: пока панель лежит, `Gate` продолжает работать по последнему `authz.json`. Это сознательная отказоустойчивость (см. Решение 4).
- Упрощает масштабирование: «доставить каждому серверу его срез `authz.json`» — единственная задача координации (см. Решение 3).

#### Решение 3. Горизонтальное масштабирование проектируется с Этапа 1 (а не «когда-нибудь в v2.0»)

**Решение:** долгосрочная цель — **одна центральная панель (источник истины в SQLite) + N дешёвых серверов OlcRTC** (50–100 клиентов на сервер, свой `server.yaml`/`authz.json`/`Gate`). Клиент закрепляется за сервером через `server_id`. Панель не маршрутизирует трафик — она **доставляет каждому серверу его срез allowlist** и следит, что он применился (`MultiServerAuthzCoordinator`, `ServerRegistry`, `ServerAuthzPusher`, `ServerHealthService`).

**Обоснование (исправление приоритизации по ревью):** даже если реальный мульти-push включается позже, **контракт координации** (`servers`, `server_id` в `subscriptions`/`user_devices`, версионирование payload, атомарность записи) закладывается в схему **с Этапа 1**. Иначе придётся ломать модель данных и `authz_writer` на ходу. Подробно — раздел 4 в [[OlcPanel_Анализ_Marzban_3xui]] и [[OlcPanel_07_Инфраструктура]].

#### Решение 4. Поведение `Gate` при сбое файла — **last-known-good** (а не fail-open) ✅ ЗАФИКСИРОВАНО

**Проблема:** ранний `Gate` при отсутствии/порче `authz.json` работал **fail-open** — `Allowed()` возвращал `true` для всех. Для *доступности* это верно, но для *платного продукта* это **дыра выручки**: самый частый класс отказов (частичная запись, пропажа файла, ошибка диска, неудачный push) тихо превращался в бесплатную раздачу VPN всем, включая неоплативших и заблокированных.

**Решение владельца (14.06.2026):** в paid-режиме `Gate` работает по схеме **last-known-good (LKG)** — `authz.fail_mode: lkg`. При ошибке свежего чтения/парсинга `Gate` **продолжает применять последний успешно загруженный и провалидированный allowlist**, а не открывается всем и не закрывает всех. Fail-open остаётся только как явный диагностический режим (`authz.fail_mode: open`) для отладки и free-инсталляций.

**Обоснование:**
- Закрывает дыру выручки, **не наказывая платящих**: сбой файла = no-op, действует последнее корректное состояние.
- Логически продолжает Решение 2: «упала панель — сервер живёт по последнему файлу» дополняется «побился файл — `Gate` живёт по последнему валидному состоянию в памяти». Один принцип «деградируй до последнего хорошего, не до катастрофы».
- Дешевле и безопаснее grace-таймера (fail-closed): не требует подбора таймаута, корректен и при секундном, и при многочасовом сбое.

**Следствия для других документов:** дизайн `Gate` (поля LKG, валидация, `lkg_max_age`, cold-start policy) — [[OlcPanel_03_Архитектура_сервисов]]; операционные процедуры рестарта/доставки — [[OlcPanel_07_Инфраструктура]]; модель угроз «потеря файла ≠ открытый доступ» и остаточный риск «устаревший allowlist» — [[OlcPanel_08_Безопасность]]; обоснование выбора целиком — раздел 5.2 в [[OlcPanel_Анализ_Marzban_3xui]].

#### Решение 5. Трафиковый аккаунтинг де-приоритизирован

**Было (v1.x):** учёт трафика на клиента (iptables MARK + counters) как часть ядра, продажа «по объёму».

**Стало (v2.0):** WebRTC/Jitsi-туннель **не отдаёт пер-клиентскую статистику** (нет Xray-stats, идентичность — `deviceId`, а не пара inbound+ключ). Продажа — **по сроку**, не по объёму. Таблицы `traffic_*` остаются «спящими» (заложены в схему, но не наполняются), на случай если позже появится серверная телеметрия. Подробности — [[OlcPanel_Анализ_Marzban_3xui]] (раздел 6, «что нельзя внедрять»).

### 0.3 Что такое OlcPanel (актуальное определение)

**OlcPanel** — веб-приложение (панель администратора) для управления клиентами, подписками и устройствами в системе OlcRTC. На старте размещается на том же VPS, что и сервер OlcRTC; при масштабировании выносится в центральную панель над флотом серверов.

```
┌─────────────────────────────────────────────────────────────────┐
│                          OlcPanel v2.0                           │
├─────────────────────────────────────────────────────────────────┤
│  👤 Клиенты и подписки                                            │
│     • Создание/редактирование/блокировка клиентов (soft-block)    │
│     • Тариф + срок подписки; продление; история                   │
│                                                                   │
│  🔑 Доступ и устройства                                           │
│     • Привязка устройств клиента (deviceId / x-hwid)              │
│     • Генерация ссылки подписки для olcbox (olcrtc:// URI)        │
│     • Авто-обновление authz.json при любом изменении доступа       │
│                                                                   │
│  🌐 Флот серверов (фундамент с Этапа 1)                           │
│     • Реестр серверов (server_id, статус, health)                 │
│     • Доставка среза allowlist на нужный сервер                    │
│     • Контроль применения: applied_version / lkg_valid_at         │
│                                                                   │
│  💳 Монетизация                                                   │
│     • Тарифы; ручная отметка оплаты; авто-блок по сроку           │
│     • Напоминания об истечении за N дней                          │
│                                                                   │
│  🛡️ Безопасность и эксплуатация                                  │
│     • JWT + bcrypt; (на multi-server) 2FA/секретный путь/fail2ban │
│     • LKG-гарантия: сбой authz.json ≠ открытый доступ             │
│     • HTTPS (Let's Encrypt); авто-бэкап БД                        │
└─────────────────────────────────────────────────────────────────┘
```

### 0.4 Стек технологий и обоснование выбора

```
┌────────────────┬────────────────────────────────────────────────────┐
│ Слой           │ Технология                                          │
├────────────────┼────────────────────────────────────────────────────┤
│ Backend        │ Python 3.11 + FastAPI + Uvicorn                     │
│ База данных    │ SQLite 3 (SQLAlchemy 2.0 + Alembic)                │
│ Frontend       │ Vanilla HTML5 + CSS3 + JS ES2022                    │
│ Веб-сервер     │ Nginx (reverse proxy + статика)                    │
│ HTTPS          │ Let's Encrypt (Certbot auto-renew)                 │
│ Процессы       │ systemd (olcrtc.service + olcpanel.service)        │
│ Аутентификация │ JWT (python-jose) + bcrypt (passlib) [+ fp-инвалид.]│
│ Планировщик    │ APScheduler (в процессе FastAPI)                   │
│ Сервер доступа │ olcrtc (Go 1.26) + пакет internal/authz (Gate)     │
│ Контракт authz │ authz.json (панель пишет, Gate читает)             │
└────────────────┴────────────────────────────────────────────────────┘
```

**Python + FastAPI (а не Node.js/Go для панели):** удобная работа с системой (subprocess, файлы), авто-Swagger (открывает путь Telegram-боту и `/internal/authz/update`), зрелая связка SQLAlchemy+Alembic, Pydantic-валидация. Сам сервер доступа — на Go (это OlcRTC), панель его не переписывает, а интегрируется через файл-контракт.

**SQLite (а не PostgreSQL):** 0 МБ доп. RAM, бэкап = `cp` файла, более чем достаточно на десятки-сотни клиентов. **Важно для multi-server:** SQLite остаётся единственным источником истины, из которого пересчитывается срез `authz.json` для каждого сервера, поэтому бэкап БД обязателен ([[OlcPanel_07_Инфраструктура]]). При росте — переключение на PostgreSQL заменой connection string без изменения кода.

**Vanilla JS (а не React/Vue):** нет сборки/npm, ~мало кода, мгновенная загрузка через Nginx, достаточно для CRUD + дашборда флота (Chart.js по CDN).

**systemd (а не Docker):** на VPS с ~1.9 GB RAM Docker даёт оверхед; systemd даёт автозапуск, journald-логи, restart-on-failure, resource limits. Управляет и `olcrtc.service`, и `olcpanel.service`.

**APScheduler (а не голый Cron):** встраивается в процесс FastAPI, имеет доступ к БД/логгеру; на нём строится авто-блок по сроку и напоминания (см. [[OlcPanel_05_Бизнес_логика]]).

### 0.5 Scope текущей версии (MVP) и явные границы

**Входит в v2.0 (MVP, Этап 1):**
- Базовая модель данных: `clients/users`, `subscriptions`, `user_devices(device_id, server_id, last_used_at)`, `servers`, `plans`, `audit_log`.
- Создание клиентов и подписок; привязка устройств; генерация ссылки подписки.
- Интеграция Phase 1: `authz_writer` + **атомарная запись** + монотонный `version` в payload.
- **Фундамент координации** (`ServerRegistry`, `Coordinator` + `LocalFilePusher` как no-op для одного сервера).
- Авто-блок по сроку (Cron→`authz.json`), базовый аудит.
- `Gate` в режиме `fail_mode: lkg` (Решение 4).

**Откладывается (с указанием этапа):**

| Функциональность | Причина отсрочки | Этап |
|---|---|---|
| Реальная мульти-серверная доставка (push A→B) | Сначала контракт и один сервер | Этап 2–3 |
| Telegram-уведомления (вкл. push-failure) | Не критично на старте | Этап 2 |
| 2FA / секретный путь / fail2ban | На multi-server критичнее | Этап 2 (раньше при флоте) |
| Авто-бэкап в Telegram, мониторинг флота | После базовой доставки | Этап 2–3 |
| Клиентский личный кабинет, автоплатежи | Фокус на admin-панели | Этап 3 |
| Онлайн/активные устройства | Нужна телеметрия сервера | Этап 3 |

Это **осознанные** границы, а не недоработки. Приоритеты и оценки — раздел 7–8 в [[OlcPanel_Анализ_Marzban_3xui]].

### 0.8 Глоссарий (актуализирован под v2.0)

| Термин | Определение |
|---|---|
| **OlcRTC** | Прокси-туннель поверх WebRTC (SCTP DataChannel) для обхода DPI. Сервер на Go. |
| **OlcBox** | Клиентское приложение (Android/Desktop) для подключения к OlcRTC; шлёт `deviceId` (`x-hwid`). |
| **OlcPanel** | Данная панель управления клиентами/подписками/устройствами. |
| **deviceId / x-hwid** | Идентификатор устройства клиента; **основная единица идентичности** в v2.0 (вместо «комната+ключ»). |
| **authz.json** | Файл-контракт `{version, updated_at, allow[], deny[]}`; панель пишет, `Gate` читает. |
| **Gate** | Серверный компонент (`internal/authz/authz.go`), `Allowed(deviceID)` решает доступ на handshake. |
| **fail_mode** | Режим `Gate` при сбое файла: `lkg` (прод) / `open` (диагностика) / `closed`. |
| **last-known-good (LKG)** | Последний валидный allowlist в памяти `Gate`; применяется при ошибке чтения свежего файла. |
| **Phase 1** | Реализованная база: панель пишет `authz.json` → сервер блокирует/пускает по `deviceId`. |
| **server_id** | Привязка клиента/устройства к конкретному серверу флота (фундамент масштабирования). |
| **Coordinator / Pusher / Registry / Health** | Сервисы координации authz для multi-server (см. 03). |
| **Общая комната + общий ключ** | Одна Jitsi-комната и один `crypto.key` на сервер; различие клиентов — по `deviceId`. |
| **ТСПУ / DPI** | Система глубокой инспекции трафика РФ / технология анализа пакетов. |
| **Admin** | Владелец сервиса; на Этапе 3 возможны ограниченные админы/реселлеры (RBAC). |

### 0.11 Принципы проектирования (в порядке приоритета)

1. **Работоспособность прежде всего.** Работающая система важнее красоты. SQLite + один файл вместо микросервисов — пока этого хватает.
2. **Эволюционность.** Каждое решение допускает постепенное развитие без переписывания — особенно модель данных и механизм авторизации (`server_id` закладывается заранее, см. Решение 3).
3. **Деградация до последнего хорошего, не до катастрофы.** Сбой инфраструктуры не должен ни рубить платящих, ни раздавать доступ бесплатно: панель упала → сервер по последнему файлу; файл побился → `Gate` по LKG (Решение 4).
4. **Явная интеграция с Phase 1.** `authz.json` + `Gate` — фундамент, а не временное решение. Любой новый компонент учитывает контракт файла, атомарность и `version`.
5. **Минимально достаточная сложность.** Не вводим абстракции (общий сетевой storage, очереди) раньше реальной необходимости; файловый контракт выбран ради автономности серверов.
6. **Горизонтальная масштабируемость как цель.** Даже на MVP закладываем переход к флоту серверов.
7. **Прозрачность и наблюдаемость.** Каждое архитектурное решение обосновано здесь; каждая операция координатора пишется в аудит; свежесть applied-authz видна на дашборде.
